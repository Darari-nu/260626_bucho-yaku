## ユーザー導線
1. ユーザーが技術記事のURLから抜粋した文章、社内共有メモ、ニュース本文、企画メモを入力欄に貼り付ける。
2. 「部長訳する」ボタンを押すと、JSが辞書に登録されたカタカナ語・IT/AI語を検出し、JTC語へ置換する。同時に3層出力を生成する。
3. ユーザーは「部長向け1行サマリー」「稟議に書ける効果文」「想定ツッコミ3つと先回り回答」をコピーし、Slack、Teams、メール、稟議下書きに貼り付ける。

## サイトマップ
1ページ内の論理セクションは次の通り。

1. ヒーロー: 「カタカナの未来を、部長に通る言葉へ。」という価値提示。
2. 入力セクション: テキスト貼り付け欄、サンプル入力、変換ボタン。
3. 翻訳結果セクション: 3層出力、置換された語の一覧、コピー操作。
4. 辞書一覧セクション: カテゴリ別の元語・部長訳・例文カード。
5. 辞書追加カスタムセクション: ユーザーがその場で追加できるローカル辞書入力。
6. 注記セクション: 私的解釈であること、外部送信しないこと、業務利用は各自判断であること。

## 公開ページ
公開ページはトップ1ページのみ。URLはGitHub Pagesのルートを想定する。

### セクション構成
- 入力欄: 複数行テキストエリア。記事本文、メモ、会議資料の一部を貼れる。サンプル文を入れるボタンを用意する。
- 翻訳結果: 上部に3層出力を表示する。各ブロックにコピーボタンを付ける。
- 辞書一覧: `AI`、`組織`、`手法`、`稟議`などのカテゴリで絞り込めるカード一覧。
- 辞書追加カスタム: 元の語、部長訳、例文、カテゴリを入力し、現在のブラウザ上の変換に反映する。v1ではセッション中のみ、v4でlocalStorage保存。

## 管理画面
管理画面は作らない。辞書の公式データは運用者が`data/dictionary.json`を編集してGitで更新する。

ユーザー向けには管理画面ではなく、公開ページ内の「辞書追加カスタム」を用意する。これは個人の会議・部署向けに言い換えを足すための簡易機能であり、サーバー保存や公開共有はしない。

## データ設計
辞書データは`data/dictionary.json`で管理する。初期版はシンプルなオブジェクト構造にし、Vanilla JSで読みやすくする。

```json
{
  "DX": {
    "translation": "デジタル化",
    "summaryPhrase": "紙や人手の仕事を、後から確認できる形に変える取り組み",
    "effect": "作業の属人化を減らし、確認漏れや二重入力を減らせます。",
    "category": "組織",
    "example": "DXを進める → 紙とExcel中心の業務をデジタル化して、確認漏れを減らす",
    "objections": [
      {
        "question": "今のやり方で困っているのか",
        "answer": "現状でも回っていますが、担当者依存と確認漏れが残るため、まず小さい範囲で効果を見ます。"
      }
    ]
  },
  "AIエージェント": {
    "translation": "担当者の一次切り分け支援",
    "summaryPhrase": "人が判断する前の確認作業を手伝う仕組み",
    "effect": "問い合わせや確認作業の初動を揃え、担当者ごとの差を減らせます。",
    "category": "AI",
    "example": "AIエージェントを導入する → 担当者の一次切り分け支援を入れる",
    "objections": []
  }
}
```

### 必須フィールド
- `translation`: 部長訳。画面上で最も強く見せる和語。
- `summaryPhrase`: 1行サマリー生成に使う説明。
- `effect`: 稟議向け効果文に使う文。
- `category`: `AI`、`組織`、`手法`、`業務改善`、`稟議`など。
- `example`: 使い方が分かる変換例。
- `objections`: 想定ツッコミと回答。語ごとの専用回答がない場合は共通テンプレートを使う。

### 初期辞書の原液
- `DX` → `デジタル化`
- `アジャイル` → `小さく試す`
- `PoC` → `お試し導入`
- `AIエージェント` → `担当者の一次切り分け支援`
- `自動化` → `確認漏れを減らす`
- `生産性向上` → `属人化の解消`

darariがJTCで実際に通してきた言い換えを優先し、単語として正確かどうかより「部長が判断しやすい言葉か」を採用基準にする。

## 業務ロジック
### 辞書マッチング
- 入力テキストをそのまま解析し、辞書キーに一致する語を検出する。
- 大文字小文字の揺れがある語は正規化する。例: `poc`、`PoC`、`POC`は`PoC`として扱う。
- 長い語を優先して置換する。例: `AIエージェント`を先に処理し、その後に`AI`を処理する。
- 同じ語が複数回出ても、置換語一覧では1回にまとめる。
- 未登録語は消さずに残す。ユーザーに「辞書未登録」として追加候補を示す。

### 3層出力生成
1. 部長向け1行サマリー
   - 検出された主要語の`summaryPhrase`を最大2つ使い、「つまり、○○を小さく試し、△△を減らす話です。」の形に整える。
   - 技術名ではなく、業務上の変化を主語にする。
2. 稟議に書ける効果文
   - `effect`を最大3つ組み合わせる。
   - 優先語は`属人化の解消`、`確認漏れを減らす`、`一次切り分け支援`、`お試し導入`の順。
   - 効果は「削減」「効率化」だけでなく、「確認しやすい」「引き継ぎやすい」「小さく検証できる」のようなJTCで通る表現にする。
3. 想定ツッコミ3つと先回り回答
   - 語ごとの`objections`を優先する。
   - 足りない場合は共通ツッコミを使う。
   - 共通ツッコミは「費用対効果は」「現場が使えるのか」「責任所在はどうするのか」。

### 優先順位
1. ユーザーが追加したカスタム辞書
2. 公式辞書の長い語句
3. 公式辞書の短い語句
4. 共通テンプレート

このロジックは「悩みをそのまま機能にする」のではなく、ユーザーが社内で止められる前に、止められる理由を先に言葉へ変えるための発見を提供する。

## 外部サービス連携
なし。翻訳API、生成AI API、アクセス解析、広告タグ、問い合わせフォーム、CRM連携は初期版に入れない。

理由は、サイトの信頼を「入力した社内っぽい文章が外へ出ない」ことに置くため。GitHub Pagesで静的に配信し、処理はユーザーのブラウザ内だけで完結させる。

## セキュリティ
- 入力テキストは外部送信しない。
- サーバー保存、ログ保存、Cookieによる追跡を行わない。
- `innerHTML`へ入力値を直接流し込まず、`textContent`で描画する。
- 辞書追加カスタムでも、ユーザー入力はHTMLとして解釈しない。
- コピー機能はユーザー操作時のみ実行する。
- 将来localStorageを使う場合も、保存対象はユーザー追加辞書のみとし、貼り付け本文は保存しない。

## インフラ
GitHub Pagesを利用する。構成は静的ファイルのみ。

```text
/
  index.html
  style.css
  script.js
  data/dictionary.json
  docs/
```

Node.js、React、ビルドツール、サーバーは不要。GitHub PagesはHTML/CSS/JavaScriptをリポジトリから公開できるため、コンテスト締切までの速度と保守性を優先する。

## デプロイ
`main`ブランチへのpushを公開元にする。GitHub Pagesの設定で`Deploy from a branch`を選び、`main`ブランチの`/root`を指定する。

将来GitHub Actionsを使う場合も、ビルドなしでリポジトリ全体をPagesへアップロードするだけに留める。静的サイトとして壊れにくい運用を優先する。

### 注記内容

## テスト
手動テストを中心に行う。ブラウザは最新版のChrome、Safari、Edgeで確認する。

### 基本用語の置換
- `DXを進めたい` → `デジタル化を進めたい`が表示される。
- `アジャイルにPoCする` → `小さく試す`、`お試し導入`が検出される。
- `AIエージェントで自動化する` → `担当者の一次切り分け支援`、`確認漏れを減らす`が出る。

### 3層出力
- 部長向け1行サマリーが技術名だけで終わらない。
- 稟議に書ける効果文に`属人化の解消`や`確認漏れを減らす`が自然に入る。
- 想定ツッコミが3つ表示され、回答が反論ではなく先回り説明になっている。

### 未登録語の挙動
- 未登録語は本文から消えない。
- 未登録語があってもエラーにならない。
- 追加候補としてユーザーが辞書追加できる。

### ユーザー追加辞書
- 入力したカスタム語が公式辞書より優先される。
- HTMLタグのような文字を入れても画面が崩れない。
- リロード時の扱いが仕様どおりである。v1は消えてよい、v4はlocalStorageに残る。

- ユーザー追加辞書をlocalStorageに保存。

- インポート・エクスポート用JSONを用意し、外部サーバー保存なしで共有できるようにする。
